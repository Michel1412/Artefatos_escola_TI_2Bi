CONTEXTO DO PROJETO

O projeto se chama "LogViewer". É um sistema SaaS multiplataforma (Web e Mobile) projetado para varredura, captura, análise inteligente e gestão de logs de erros de múltiplas aplicações. O sistema utiliza um LLM para cruzar dados de erros com o contexto específico da empresa do cliente, categorizar falhas e sugerir soluções precisas.

STACK TECNOLÓGICA E ARQUITETURA

  • Arquitetura: Hexagonal (separação estrita entre Core Domain, Ports e Adapters).

  • Frontend Web: React (ou React Native Web) otimizado para baixo PLT.

  • Frontend Mobile: React Native (Android e iOS, com biometria e espelho do dashboard web).

  • Banco de Dados: MongoDB (ideal para maleabilidade de logs não estruturados).

  • Autenticação: Clerk (suporte a 2FA e gestão segura de usuários).

  • SDKs: Desenvolvidos inicialmente em Java, com comunicação criptografada (túnel) e integração via Webhooks para alta performance.

MECÂNICA PRINCIPAL E FLUXO DE DADOS

  1. Captura: O sistema recebe logs via SDK instalada no cliente ou varrendo uma pasta de arquivos configurada.

  2. Inteligência Contextual: O LLM cruza o log capturado com o "Contexto" predefinido pelo cliente (Ramo de atuação, Stack, Criticidade).

  3. Persistência: O log é tipificado e salvo no MongoDB contendo: Timestamp, Tipo_do_Erro, Conteúdo_do_Log e Contexto_Aplicado.

  4. Alerta (Multicanal): Notificações assíncronas via E-mail e Push (Mobile) para emergências; e síncronas via WebSockets (Toasts/Modais no Dashboard Web).

  5. Interação LLM: O cliente acessa o log e interage com um Agente IA para buscar soluções.

REGRAS DE NEGÓCIO INEGOCIÁVEIS (CONSTRAINTS)

  • Gestão de Usuários, Planos e Sistemas

    • RN-01 (Identidade e Precificação): Todo usuário base é um CNPJ. Ele pode cadastrar múltiplos sistemas. O preço da assinatura (fixo ou negociável via API de pagamento) aumenta proporcionalmente a cada novo sistema adicionado.

    • RN-02 (Tokens e Isolamento): Os tokens de integração da SDK devem ser únicos por sistema cadastrado.

    • RN-03 (Exigência de 2FA): Usuários que ativarem a Autenticação de Dois Fatores (2FA) pelo painel Web são obrigados a ter o aplicativo mobile instalado para aprovação.

    Processamento e Inteligência Contextual

    • RN-04 (Inteligência Contextual e Tipificação): Quando um log é capturado, a IA cruza o erro com o Contexto configurado (ramo, stack, criticidade). O erro é classificado e salvo com as seguintes informações obrigatórias: Timestamp, Tipo_do_Erro, Conteúdo_do_Log e Contexto_Aplicado.

    • RN-05 (Prevenção de Dados Sensíveis): O sistema deve validar ativamente e ofuscar credenciais ou senhas vazadas nos logs antes da persistência.

    Interação com IA e Fluxo de Solução

    • RN-06 (Sugestão Passiva por Consentimento): O sistema NÃO aplica correções automaticamente. Ele deve apenas apresentar uma solução sugerida (ex: "Deseja receber uma possível solução para o erro X?") e disponibilizar o link para o ambiente do Agente.

    • RN-07 (Mudança de Status via IA): Ao solicitar/aceitar uma solução através do LLM, o status do erro muda automaticamente de "Não Solucionado" para "Solucionado".

    Notificações e Alertas

    • RN-08 (Notificação Multicanal Obrigatória): Todo erro processado segue o fluxo: Identificação → Persistência → Alerta. Alertas assíncronos vão para E-mail e Push Mobile (especialmente situações emergenciais). Alertas síncronos aparecem em tempo real via WebSockets no Dashboard Web (Toasts/Modais).

    • RN-09 (Silenciamento/Fadiga de Alerta): Se um mesmo erro ocorrer repetidamente num intervalo de 5 minutos, as notificações de e-mail devem ser agrupadas num único resumo.

    Dashboards, Filtros e Relatórios

    • RN-10 (Visões do Dashboard): O Dashboard deve exibir dados consolidados, sugestões de problemas recorrentes e possuir um filtro mestre para visualizar dados de um sistema específico ou "multi-sistemas".

    • RN-11 (Busca e Exportação): A barra de pesquisa deve buscar em todo o histórico armazenado. Os relatórios gerados a partir dessas buscas ou dashboards podem ser exportados para PDF e Excel.

    • RN-12 (Gestão de Marcadores): O usuário pode usar marcadores padrão do sistema (Solucionado, Pendente, Arquivado, Ignorado) ou criar marcadores personalizados (estilo labels de e-mail) para organizar a caixa de erros
