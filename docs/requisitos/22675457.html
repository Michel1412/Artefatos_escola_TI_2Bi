Minha equipe de software : Caso de uso - 2: Identificação de logs sem padrões estruturados

Created by João Pedro Souza Peixoto Saraiva, last modified on mar. 27, 2026

Nome: Recebimento e Aceitação de Logs Agnósticos (Sem Padrão Estruturado)

Atores:

  • Sistema Cliente (SDK) ou Serviço de Varredura (Scanner de Diretório)

  • Backend LogViewer (Porta de Entrada / API)

  • Banco de Dados MongoDB (Adapter de Persistência)

Descrição: Garante que o sistema atue como um receptor flexível, sendo capaz de identificar, aceitar e armazenar a entrada de logs de erros em sua forma bruta (raw text, arquivos variados ou payloads não padronizados), sem exigir um schema rígido (como um JSON ou XML predefinido) por parte do cliente.

Pré-condições:

  • O cliente possui um sistema cadastrado e ativo.

  • A autenticação/token de integração da origem é válida (RF-16 / RN-02).

  • A conexão entre a origem e o sistema está estabelecida de forma segura (RNF-01).

Pós-condições:

  • O log, independentemente do seu formato, é aceito pelo backend sem erros de validação de estrutura (schema validation).

  • O conteúdo bruto é armazenado com sucesso no MongoDB, aguardando a esteira de processamento inteligente.

  • O sistema de origem recebe a confirmação de que o log foi capturado.

Fluxo Principal:

  1. O Sistema Cliente (via SDK ou Varredura) envia um payload de log de erro para o endpoint de captura do LogViewer.

  2. O sistema (Backend LogViewer) recebe a requisição.

  3. O sistema realiza a validação de segurança primária (verifica a autenticidade do token e mascara dados sensíveis básicos, conforme RN-02 e RN-05).

  4. O sistema identifica o payload como um "Log de Erro" associado ao sistema de origem, ignorando a validação de estrutura interna do arquivo/texto.

  5. O sistema empacota o conteúdo recebido em um formato bruto (RawLogEntry), anexando apenas metadados básicos de origem (ID do Sistema e Timestamp de recebimento).

  6. O sistema persiste esse documento bruto em uma coleção inicial no MongoDB (aproveitando a característica schema-less do banco).

  7. O sistema retorna um HTTP Status de sucesso (ex: 202 Accepted) para a SDK/Origem, liberando a aplicação cliente rapidamente.

  8. O sistema enfileira o log bruto para o processo subsequente de inteligência e tipificação (que ocorre em background).

Fluxos Alternativos:

  • A1: Payload excede o tamanho máximo permitido

    1. O sistema identifica que o arquivo ou texto enviado ultrapassa o limite de MB configurado para uma única requisição.

    2. O sistema rejeita o recebimento.

    3. O sistema retorna um erro de Payload Too Large (413) para a SDK, orientando o descarte ou fracionamento.

  • A2: Falha na identificação da Origem (Token/Contexto não localizado)

    1. O sistema não consegue vincular o log recebido a nenhum sistema/CNPJ cadastrado.

    2. O sistema descarta o payload imediatamente para evitar sobrecarga no banco.

    3. O sistema retorna erro de autenticação para a origem.

Itens associados:

Regras de Negócio:

  • RN-02: Tokens de integração devem ser únicos por sistema.

  • RN-05: Dados sensíveis devem ser obrigatoriamente mascarados antes da persistência.

Requisitos Funcionais:

  • RF-07: O Sistema identifica os logs de erros, mesmo que sem padrões.

  • RF-16: Gerenciamento de tokens da SDK.

  • RF-25: Validação e mascaramento de credenciais.

Requisitos Não Funcionais:

  • RNF-01: Comunicação criptografada obrigatória (TLS/Túnel seguro).

  • RNF-03: Autenticação gerida via Clerk.

Observações (Notas de Arquitetura):

  • Este caso de uso reflete o padrão Fire and Forget para o cliente. A API de entrada (Inbound Adapter) deve ser extremamente rápida e permissiva quanto ao formato do corpo da requisição (text/plain, application/octet-stream, application/json desestruturado, etc.).

  • O uso do MongoDB aqui é justificado pela capacidade de salvar um documento onde o campo raw_content pode ser literalmente qualquer coisa, adiando a complexidade estrutural para o momento em que a IA (LLM) entrar em ação em um caso de uso posterior.
