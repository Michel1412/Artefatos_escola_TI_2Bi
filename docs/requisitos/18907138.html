TR-21 - Criação de caso de uso - 4: Integrar SDK e Processar Logs com Segurança Concluído

Nome: Integrar SDK e Processar Logs com Segurança

Atores:

  • Sistema Cliente (SDK)

  • Backend LogViewer

  • Provedor de Autenticação (Clerk)

Descrição:
Permite que aplicações clientes, por meio da SDK, enviem logs de forma segura, autenticada e criptografada, garantindo o mascaramento de dados sensíveis e a validação dos tokens de integração.

Pré-condições:

  • O cliente possui um sistema cadastrado

  • Um token de integração foi gerado para o sistema

  • A SDK está configurada com o token válido

  • A comunicação com o backend está configurada via canal criptografado

Pós-condições:

  • Logs recebidos são validados, sanitizados e processados

  • Dados sensíveis são mascarados antes da persistência

  • Logs são aceitos ou rejeitados com base na validade do token

Fluxo Principal:

  1. A aplicação cliente gera um log de erro

  2. A SDK captura o log automaticamente

  3. A SDK estabelece conexão segura com o backend (túnel criptografado)

  4. A SDK envia o log junto com o token de integração

  5. O sistema recebe a requisição

  6. O sistema valida o token de integração:
    6.1 Verifica se o token existe
    6.2 Verifica se o token está ativo
    6.3 Verifica associação com um sistema válido

  7. O sistema valida o payload recebido

  8. O sistema executa o processo de detecção de dados sensíveis:
    8.1 Identifica padrões de senha, tokens, credenciais
    8.2 Aplica mascaramento (ex: ****)

  9. O sistema encaminha o log para o fluxo de processamento interno

  10. O sistema retorna status de sucesso para a SDK

Fluxos Alternativos:

A1: Token inválido ou inexistente

  • O sistema rejeita a requisição

  • Retorna erro de autenticação

A2: Token revogado ou inativo

  • O sistema bloqueia o envio

  • Registra tentativa inválida

A3: Falha na conexão criptografada

  • A SDK tenta reenvio automático (retry com backoff)

  • Se persistir, registra erro local

A4: Detecção de dados sensíveis falha

  • O sistema aplica fallback de mascaramento genérico

  • Log ainda é persistido de forma segura

Regras de Negócio:

  • RN-02: Tokens de integração devem ser únicos por sistema

  • RN-05: Dados sensíveis devem ser obrigatoriamente mascarados antes da persistência

Requisitos Funcionais:

  • RF-16: Gerenciamento de tokens da SDK

  • RF-25: Validação e mascaramento de credenciais

Requisitos Não Funcionais:

  • RNF-01: Comunicação criptografada obrigatória (TLS/Túnel seguro)

  • RNF-03: Autenticação gerida via Clerk

Observações:

  • O token da SDK atua como chave de autenticação da aplicação cliente, não substituindo autenticação de usuário

  • A validação do token deve ocorrer antes de qualquer processamento do log

  • O mascaramento deve ocorrer antes de qualquer persistência ou envio ao LLM

  • Esse caso de uso deve ser implementado como um UseCase central no Core (ex: ProcessIncomingLogUseCase)

  • A comunicação da SDK deve priorizar envio assíncrono (webhooks ou fila) para não impactar a aplicação cliente
