TR-29 - Criação de caso de uso - 12: Gerenciar Erro com Sugestão de IA, Consentimento Explícito e Arquivamento Concluído

Itens associados:

Requisitos Funcionais:

  • RF-02: O Sistema deve perguntar ao Usuário se deseja uma solução para o erro descoberto.

  • RF-19: O Sistema deve sugerir soluções para problemas recorrentes.

  • RF-27: O Sistema deve permitir o arquivamento de erros.

Regras de Negócio:

  • RN-02 (Fadiga de Alertas): Se o mesmo erro ocorrer repetidamente em 5 minutos, as notificações devem ser agrupadas em um único resumo.

  • RN-01 (Sem Automação de Correção): O sistema NUNCA aplica correções automaticamente. Toda solução gerada pela IA deve passar por consentimento explícito do usuário ("Deseja receber uma possível solução?").

  • RN-05 (Mudança de Status): Ao solicitar/aceitar uma solução do LLM, o status do erro muda automaticamente para "Solucionado".

Nome: Gerenciar Erro com Sugestão de IA, Consentimento Explícito e Arquivamento

  • Atores:

    • Usuário.

    • Aplicação Web / Mobile.

    • Sistema LogViewer (Backend).

    • Serviço de LLM (IA).

    • Banco de Dados (MongoDB).

Descrição:

Permite ao usuário visualizar erros registrados, identificar recorrência, solicitar sugestões de solução baseadas em IA e gerenciar o ciclo de vida do erro (aberto, solucionado, arquivado), sempre exigindo consentimento explícito para ações da IA.

Pré-condições:

  • Usuário autenticado.

  • Erro existente no sistema.

  • Contexto do erro disponível.

  • Integração com IA ativa.

Pós-condições:

  • Sugestão de solução pode ser apresentada ao usuário.

  • Status pode ser alterado para:

    • "Solucionado".

    • "Arquivado".

  • Histórico de ações registrado.

Fluxo Principal 1 - Interação com erro Recorrente:

  1. Usuário acessa erro no dashboard.

  2. Sistema verifica recorrência:

    • count > 1 dentro da janela.

  3. Sistema exibe mensagem:
    "Esse erro já ocorreu X vezes. Deseja receber uma possível solução?".

  4. Usuário aceita solicitar solução.

  5. Sistema envia dados ao LLM:

    • log + contexto + histórico.

  6. IA retorna sugestão.

  7. Sistema apresenta solução ao usuário.

  8. Usuário decide:

    • Aceitar solução.

    • Ignorar solução.

  9. Se usuário aceitar:

    • Sistema atualiza status → "Solucionado" (RN-05).

  10. Sistema registra ação no histórico.

Fluxo Principal 2 - Erro não Recorrente:

  1. Usuário acessa erro.

  2. Sistema não sugere automaticamente.

  3. Sistema exibe opção:

    1. "Solicitar sugestão de solução".

  4. Usuário solicita.

  5. Fluxo segue a partir do passo 5 do fluxo anterior.

Fluxo Principal 3 - Arquivamento de Erro:

  1. Usuário seleciona erro.

  2. Usuário escolhe "Arquivar".

  3. Sistema altera status → "Arquivado".

  4. Sistema remove erro das listas ativas.

  5. Sistema mantém histórico para auditoria.

Fluxo Alternativo:

A1: Usuário recusa sugestão

  • Nenhuma chamada ao LLM é feita.

  • Status permanece inalterado.

A2: Usuário ignora solução após visualizar

  • Status permanece "Aberto".

A3: Falha no serviço de IA

  • Sistema retorna erro amigável.

  • Permite nova tentativa.

A4: Arquivamento direto

  • Usuário arquiva sem interação com IA.

  • Status → "Arquivado".

Regras de Negócio:

  • RN-01: Sistema nunca aplica solução automaticamente.

  • RN-05: Status só muda para "Solucionado" após aceite explícito.

  • RN-02: Recorrência deve ser considerada na UX.

  • Erros arquivados:

    • Não geram alertas.

    • Permanecem acessíveis para histórico.

  • Sugestão da IA deve ser:

    • Sempre opcional.

    • Nunca intrusiva.

Requisitos:

  • RF-02: Perguntar se deseja solução.

  • RF-19: Sugerir soluções para problemas recorrentes.

  • RF-27: Permitir arquivamento de erros.

Observações:

  • Nada a Declarar.

User --> UC_Ver
User --> UC_Pedir

' O sistema precisa da IA para a sugestão
UC_Pedir -- AI

' Tudo é registrado no Banco
UC_Ver -- DB
UC_Aceitar -- DB
UC_Arquivar -- DB

' Notas das Regras de Negócio (RNs)
note top of UC_Pedir : RN-01: NADA é automático.\nO usuário PRECISA autorizar.
note bottom of UC_Aceitar : RN-05: Status só muda\nse o usuário aceitar a solução.
note top of UC_Ver : RN-02: Se o erro for repetido,\no sistema avisa na tela.
@enduml
