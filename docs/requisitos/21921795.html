TR-28 - Criação de caso de uso - 11: Processar Logs com Contexto, Agrupamento e Sanatização Concluído

Itens associados:

Requisitos Funcionais:

  • RF-03: O Sistema deve receber o caminho da pasta de Logs da Aplicação configurada.

  • RF-09: O Sistema aplicará o contexto configurado no momento da varredura.

  • RF-22: O Sistema deve ter um contador/agrupador de logs repetidos.

Regras de Negócio:

  • RN-02: Logs repetidos em até 5 minutos devem ser agrupados

  • RN-03: Dados sensíveis devem ser obrigatoriamente mascarados antes da persistência

  • RN-04: Contexto do cliente deve ser aplicado em toda análise

Nome: Processar Logs com Contexto, Agrupamento e Sanitização

  • Atores:

    • Sistema Cliente (SDK Java).

    • Core Domain (Log Processing)

    • Serviço de Ingestão.

    • Serviço de LLM (IA).

    • Banco de Dados (MongoDB).

    • Sistema LogViewer.

Descrição:

Permite ao sistema receber logs de diferentes origens (SDK ou diretório configurado), aplicar sanitização de dados sensíveis, enriquecer com contexto do cliente e agrupar ocorrências repetidas em uma janela temporal para otimização de alertas e análise.

Pré-condições:

  • Sistema cadastrado (CNPJ válido).

  • Token SDK válido e ativo.

  • Contexto do cliente configurado.

  • Origem de logs configurada (SDK ou path válido).

  • Serviço de ingestão disponível.

Pós-condições:

  • Logs sanitizados e persistidos.

  • Logs classificados com contexto aplicado.

  • Ocorrências repetidas agrupadas com contador incremental.

  • Eventos prontos para notificação (respeitando RN-02).

Fluxo Principal 1 - Ingestão via SDK:

  1. SDK envia log via requisição segura (HTTPS + token).

  2. Adapter valida token e origem.

  3. Sistema encaminha log para o Core Domain.

  4. Core executa sanitização:

    • Remove/mascara credenciais (RN-03)

  5. Core recupera contexto do cliente.

  6. Core aplica enriquecimento contextual.

  7. Core identifica tipo do erro.

  8. Sistema consulta MongoDB por ocorrências similares:

    • Baseado em hash do erro + contexto + janela de tempo (5 min).

  9. Se erro já existir na janela:

    • Incrementa contador.

    • Atualiza timestamp da última ocorrência.

  10. Se erro for novo:

  • Cria novo documento.

  • Inicializa contador = 1.

  1. Sistema persiste no MongoDB

  2. Sistema publica evento interno:

  • LogProcessedEvent

Fluxo Principal 2 - Ingestão via Diretório:

  1. Serviço monitora diretório configurado.

  2. Detecta novo arquivo de log.

  3. Lê conteúdo do arquivo.

  4. Para cada entrada:

    • Envia para pipeline interno (igual fluxo SDK a partir do passo 3).

Fluxo Alternativo:

A1: Token inválido

  • Sistema rejeita requisição.

  • Registra tentativa inválida.

A2: Caminho de logs inválido

  • Sistema rejeita configuração.

  • Retorna erro ao usuário.

A3: Log não estruturado

  • Sistema aplica heurística + LLM.

  • Classificação probabilística do erro.

A4: Falha na sanitização

  • Log é descartado OU marcado como "não confiável".

  • Evento de segurança é gerado.

Regras de Negócio:

  • RN-02: Logs repetidos em até 5 minutos devem ser agrupados

  • RN-03: Dados sensíveis devem ser obrigatoriamente mascarados antes da persistência

  • RN-04: Contexto do cliente deve ser aplicado em toda análise

  • Agrupamento deve ser baseado em:

    • hash do erro

    • contexto

    • janela temporal

Requisitos:

  • RF-03: Receber caminho de logs configurado.

  • RF-09: Processar logs com contexto.

  • RF-22: Agrupar logs repetidos.

Observações:

  • Alterações de contexto não reprocessam logs antigos automaticamente.

  • Pode existir feature futura de reprocessamento.
