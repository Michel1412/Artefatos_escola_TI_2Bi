Caso de uso - 1: Explorar, Filtrar e Interagir com Logs via Agente IA

TR-30 - Criação de caso de uso - 13 Explorar, Filtrar e Interagir com Logs via Agente IA Concluído

Itens associados:

Regras de Negócio:

  • RN-06: O sistema NÃO aplica correções automaticamente. Deve apenas apresentar uma sugestão passiva e disponibilizar o link para o ambiente do Agente.

  • RN-07: Ao solicitar/aceitar uma solução através do LLM, o status do erro muda automaticamente de "Não Solucionado" para "Solucionado".

  • RN-10: O Dashboard deve exibir dados com um filtro mestre para visualizar dados de um sistema específico ou "multi-sistemas".


Requisitos Funcionais:

  • RF-14: O Sistema deve permitir filtrar logs por múltiplas opções (ex: data, criticidade, sistema).

  • RF-21: O Sistema deve ter uma barra de pesquisa global para os erros já armazenados.

  • RF-23: O Sistema deve categorizar erros por Status (Ex: Não Solucionado, Solucionado).

Requisitos Não Funcionais:

  • RNF-04: O banco de dados principal será o MongoDB (maleabilidade para logs não estruturados).

  • RNF-06: O MongoDB utilizará um Índice TTL (Time-To-Live) configurado para expirar os documentos automaticamente após 7 dias, garantindo a retenção temporária sem necessidade de rotinas em background.

  • RNF-09: O desenvolvimento deve ser estritamente orientado a Padrões de Projeto (Arquitetura Hexagonal).

Atores:

  • Usuário

  • Aplicação Web / Aplicativo Mobile

  • Sistema LogViewer (Backend)

  • Banco de Dados (MongoDB)

  • Agente IA (LLM)

Descrição:
Permite ao usuário consultar o histórico de logs armazenados, realizando buscas textuais globais e aplicando filtros combinados. O usuário pode visualizar os detalhes de um erro específico e interagir com um Agente IA para buscar sugestões de resolução, o que altera automaticamente a categorização de status do log no sistema.


Pré-condições:

  • Usuário autenticado no sistema (Web ou Mobile).

  • O CNPJ do usuário possui sistemas cadastrados e logs indexados no MongoDB.

  • O serviço do Agente IA (LLM) está acessível e configurado.


Pós-condições:

  • Lista de logs é renderizada no Dashboard de acordo com os filtros.

  • Interações com o Agente IA são concluídas com sucesso.

  • O status do log interagido é atualizado para "Solucionado" no banco de dados.


FLUXO PRINCIPAL 1: BUSCA E FILTRAGEM DE LOGS (DASHBOARD):

  • Usuário acessa a área de Dashboard/Histórico de Erros.

  • Sistema (Frontend) solicita a listagem inicial ao Backend.

  • Backend consulta o MongoDB e retorna os logs recentes.

  • Sistema exibe a lista de erros padrão.

  • Usuário insere um termo na barra de pesquisa global OU seleciona filtros (data, criticidade, sistema específico).

  • Sistema envia os novos parâmetros de busca ao Backend.

  • Backend executa a query no MongoDB (utilizando Text Indexes para a busca global).

  • Backend retorna o conjunto de logs filtrados e paginados.

  • Sistema exibe os resultados atualizados para o Usuário.


FLUXO PRINCIPAL 2: INTERAÇÃO COM AGENTE IA E RESOLUÇÃO DO ERRO:

  • Usuário clica em um erro específico na listagem

  • Sistema exibe os detalhes completos (Conteúdo do Log, Timestamp, Contexto Aplicado)

  • Sistema exibe passivamente a mensagem: "Deseja receber uma possível solução para o erro X?"

  • Usuário clica no botão/link para acionar o Agente IA

  • Backend recebe a requisição de ajuda

  • Backend envia o payload do log e o contexto da empresa para o Agente IA (via Adapter)

  • Agente IA processa as informações e devolve a sugestão de solução

  • Backend executa a mudança de estado do log para "Solucionado" (Regra RN-07)

  • Backend persiste a alteração de status no MongoDB

  • Backend retorna a sugestão de solução para o Frontend

  • Sistema exibe a solução sugerida e atualiza a interface mostrando o novo status


FLUXOS ALTERNATIVOS:

  • A1: Nenhum log encontrado na busca/filtro:

    • Backend executa a consulta no MongoDB e não encontra documentos

    • Sistema exibe interface de "Empty State" (Nenhum erro encontrado com estes critérios)

  • A2: Usuário não deseja acionar a IA:

    • Usuário visualiza o log, mas ignora a sugestão passiva (não clica no link do Agente)

    • O status do log permanece inalterado ("Não Solucionado")

  • A3: Falha na comunicação com o Agente IA (LLM):

    • Backend tenta enviar os dados ao Agente IA e ocorre um timeout/erro

    • Sistema exibe mensagem amigável de indisponibilidade da IA

    • O status do log não sofre alteração


OBSERVAÇÕES:

  • A performance da barra de pesquisa global (RF-21) é garantida pelo uso de índices de texto compostos no MongoDB, evitando gargalos no carregamento da aplicação (baixo PLT).

  • A alteração de status (RN-07) ocorre como uma transação de domínio (Core) logo após o retorno positivo da IA, garantindo que o banco de dados e a interface fiquem sempre sincronizados.

  • Os Driving Adapters (React Web e React Native) consomem a exata mesma porta de entrada para buscar logs, garantindo que a regra de exibição multi-sistemas funcione de forma idêntica em ambas as plataformas.
