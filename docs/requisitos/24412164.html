Minha equipe de software : Caso de uso - 19: Consultar Visão Gerencial e Métricas de Sistemas (Mobile)

Created by João Pedro Souza Peixoto Saraiva on mar. 30, 2026

Itens associados:

  • Regras de Negócio:

    • RN-01: Identidade baseada no CNPJ (o dashboard deve consolidar dados de todos os sistemas atrelados à conta ou filtrar por um específico).

    • RN-10: O Dashboard deve exibir dados consolidados e possuir um filtro mestre para visualização "multi-sistemas" ou sistema individual.

  • Requisitos Funcionais:

    • RF-29: O Sistema deve possuir uma interface mobile (aplicativo).

    • RF-32: O aplicativo deve possuir um espelho simplificado do dashboard do sistema web.

    • RF-33: O aplicativo deve ser lançado para Android e iOS.

  • Requisitos Não Funcionais:

    • RNF-01: O aplicativo deve ser desenvolvido em React Native (código base único para Android e iOS).

    • RNF-02: A busca de dados deve passar por uma Driving Port (API REST/GraphQL) que se comunica com o Core Domain (Arquitetura Hexagonal).

Nome: Visualizar Dashboard Simplificado no Aplicativo Mobile

Atores:

  • Usuário Autenticado (Cliente CNPJ)

  • Aplicativo Mobile (React Native - Android/iOS)

  • Sistema LogViewer (Backend / API de Leitura)

  • Banco de Dados (MongoDB)

Descrição: Permite ao usuário acessar, através de seus dispositivos móveis (Android e iOS), uma visão gerencial e consolidada do status de seus sistemas. Este dashboard atua como um "espelho simplificado" da versão Web, focando em métricas essenciais (erros críticos recentes, taxa de resolução e alertas ativos), permitindo rápida tomada de decisão sem a complexidade da interface de análise profunda da versão desktop.

Pré-condições:

  • Usuário possui o aplicativo LogViewer instalado em seu dispositivo Android ou iOS.

  • Usuário está com uma sessão válida e ativa (já passou pelo fluxo de autenticação do Clerk, incluindo biometria, se configurado).

  • Usuário possui ao menos um sistema (API/App) cadastrado em sua conta (CNPJ).

Pós-condições:

  • Métricas essenciais de logs são carregadas e renderizadas no aplicativo.

  • Conexão via WebSockets/Polling (se aplicável para real-time) é estabelecida para atualização de contadores em tela.

FLUXO PRINCIPAL 1: ACESSO E RENDERIZAÇÃO DO DASHBOARD MULTI-SISTEMAS (PADRÃO)

  1. O Usuário abre o Aplicativo Mobile.

  2. O Aplicativo valida a sessão atual localmente e exibe a tela inicial (Dashboard).

  3. O Aplicativo Mobile envia uma requisição de leitura (GET) para o Backend solicitando os dados do dashboard, passando o filtro padrão "multi-sistemas".

  4. O Backend (Adapter de Entrada/Controller) recebe a requisição e encaminha para o Core Domain através de uma Driving Port (ex: GetDashboardMetricsUseCase).

  5. O Core Domain invoca a Driven Port (Repository) para buscar os dados consolidados.

  6. O Adapter do MongoDB executa uma Aggregation Pipeline filtrando pelo CNPJ do usuário para extrair:

    • Contagem de logs totais (últimas 24h).

    • Contador de logs por Status (Não solucionado / Solucionado).

    • Lista dos últimos 5 erros de criticidade alta (Contexto).

  7. O Backend retorna o payload estruturado (JSON) para o Aplicativo Mobile.

  8. O Aplicativo Mobile (React Native) renderiza a interface simplificada, exibindo cards de resumo e gráficos nativos otimizados.

FLUXO PRINCIPAL 2: FILTRAGEM DE DASHBOARD POR SISTEMA ESPECÍFICO

  1. A partir do Dashboard carregado, o Usuário toca no seletor de "Filtro de Sistemas" (Dropdown/BottomSheet).

  2. O Aplicativo lista os sistemas cadastrados atrelados àquele CNPJ.

  3. O Usuário seleciona um sistema específico.

  4. O Aplicativo Mobile dispara uma nova requisição ao Backend, incluindo o systemId no parâmetro de busca.

  5. O Backend repete o fluxo de busca no MongoDB, agora adicionando o systemId no $match da agregação.

  6. O Backend retorna as métricas atualizadas.

  7. O Aplicativo Mobile atualiza a interface (re-render no React Native) refletindo apenas os dados do sistema escolhido.

FLUXOS ALTERNATIVOS

  • A1: Conta sem sistemas cadastrados

    • No passo 5 do Fluxo Principal 1, o banco retorna uma lista vazia.

    • O Aplicativo Mobile renderiza um Empty State (Interface de estado vazio) informando: "Nenhum sistema cadastrado ainda", com um botão orientando o usuário a acessar a plataforma Web para registrar seu primeiro sistema e gerar os tokens da SDK.

  • A2: Falha de conexão com a Internet

    • O Aplicativo falha ao tentar buscar os dados (Timeout ou Network Error).

    • O Aplicativo exibe um Toast ou Modal de erro: "Sem conexão. Exibindo os últimos dados armazenados" e carrega o último estado salvo em cache local (AsyncStorage/MMKV), se disponível.

  • A3: Sessão inválida / Expirada

    • O Backend retorna HTTP 401 Unauthorized ao receber a requisição.

    • O Aplicativo limpa o estado local e redireciona o usuário para a tela de Login (ativando o fluxo do Clerk).

REGRAS DE NEGÓCIO

  • RN-01: Os dados devem ser estritamente isolados e filtrados com base no CNPJ atrelado ao usuário autenticado.

  • RN-10: A visão padrão do mobile deve ser "multi-sistemas", com a opção de o usuário realizar o drill-down para um sistema específico através de um filtro mestre na interface.

REQUISITOS FUNCIONAIS

  • RF-29 / RF-33: O desenvolvimento desta tela deve utilizar componentes compatíveis com compilação cross-platform para garantir o funcionamento idêntico no Android e iOS.

  • RF-32: A interface não deve conter tabelas complexas ou logs textuais completos (como na Web), focando em Cards de contadores e visualização rápida (Espelho Simplificado).

REQUISITOS NÃO FUNCIONAIS

  • A latência da requisição do Dashboard deve ser mínima, delegando a carga de processamento das métricas para as Aggregation Pipelines do MongoDB, evitando processamento em memória no Node.js/Java.

OBSERVAÇÕES

  • Arquitetura Hexagonal: O Aplicativo Mobile interage apenas com a borda externa da aplicação (Adapters). O Core Domain não sabe que os dados estão sendo entregues para um app React Native; ele apenas devolve os DTOs com as métricas calculadas.

  • Design Mobile (React Native): Recomenda-se o uso de bibliotecas de gráficos otimizadas para mobile (como react-native-skia ou react-native-svg) para garantir que os gráficos do dashboard tenham 60fps constantes, mantendo a sensação de fluidez nativa.

  • Interação com IA: Como o dashboard mobile é um resumo, ao clicar em um erro crítico da lista de "Top 5 Erros", o usuário deve ser levado a uma tela de detalhes onde o RF-31 (facilitar a comunicação com o LLM) entra em ação.
