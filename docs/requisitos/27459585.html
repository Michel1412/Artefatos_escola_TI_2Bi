Minha equipe de software : Caso de uso - 17: Gerenciamento de Assinaturas e Planos (Billing)

Created by Vinicius Ferrarini, last modified by Peng on mai. 04, 2026

Regras de Negócio:

  • RN-01 (Identidade e Precificação): Todo usuário base é um CNPJ. O valor da assinatura é composto por uma base fixa ou negociável, acrescida proporcionalmente a cada novo sistema cadastrado no painel.

  • RN-02 (Tokens e Isolamento): A ativação de funcionalidades avançadas ou novos tokens de sistema depende do status ativo da assinatura.

Requisitos Funcionais:

  • RF-18: O Sistema deve conter um gerenciamento de assinatura e planos do usuário.

  • RF-28: O Sistema deve ter uma integração com uma API de pagamento (ex: Mercado Pago, Abacate Pay).

  • RF-15: O Sistema deve gerir o Usuário (atrelado ao fluxo de conta/plano).

Requisitos Não Funcionais:

  • RNF-04: O banco de dados principal será o MongoDB (armazenamento de histórico de faturas e status do plano).

  • RNF-09: Desenvolvimento orientado a Padrões de Projeto (Uso de Adapter Pattern para isolar a API de pagamento).

Nome: Gerenciamento de Assinaturas e Planos (Billing)

Atores:

  • Usuário Gestor (Dono do CNPJ)

  • Sistema LogViewer (Backend)

  • Provedor de Pagamento (Gateway Externo - ex: Mercado Pago)

  • Banco de Dados (MongoDB)

Descrição: Permite ao gestor da conta visualizar seu plano atual, realizar o upgrade/downgrade de assinaturas, adicionar novos sistemas (o que impacta o valor final) e realizar o pagamento através de gateways externos. O sistema controla o ciclo de vida da assinatura (Ativa, Pendente, Cancelada) para liberar ou bloquear recursos da plataforma.

Pré-condições:

  • Usuário autenticado e com permissão de administrador do CNPJ.

  • Conexão estável com o gateway de pagamento externo.

Pós-condições:

  • Status da assinatura atualizado no MongoDB.

  • Limites de cadastro de novos sistemas ajustados conforme o plano contratado.

  • Recibo/Confirmação gerado para o usuário.

FLUXO PRINCIPAL 1: CONTRATAÇÃO OU UPGRADE DE PLANO:

  1. O Usuário acessa a aba "Assinatura e Planos".

  2. O Sistema exibe o plano atual e as opções de escala (preço por sistema adicional - RN-01).

  3. O Usuário seleciona o plano desejado ou adiciona a quantidade de novos sistemas que deseja monitorar.

  4. O Sistema calcula o valor total e solicita o método de pagamento.

  5. O Usuário confirma o pagamento.

  6. O Backend aciona o Adapter de Pagamento enviando os dados da transação para o Gateway Externo.

  7. O Gateway processa o pagamento e retorna o status (Aprovado).

  8. O Backend atualiza a validade da assinatura e os limites de sistemas no MongoDB.

  9. O Sistema exibe mensagem de sucesso e libera o acesso aos recursos.

FLUXO PRINCIPAL 2: RENOVAÇÃO AUTOMÁTICA E WEBHOOKS:

  1. O Gateway de Pagamento processa a recorrência mensal.

  2. O Gateway envia uma notificação assíncrona (Webhook) para o Backend do LogViewer.

  3. O Backend valida a autenticidade da notificação.

  4. O Backend estende a data de expiração do plano no banco de dados.

FLUXOS ALTERNATIVOS:

  • A1: Pagamento Recusado/Falho:

    • No Passo 7 do Fluxo 1, o Gateway retorna erro (cartão recusado, saldo insuficiente).

    • O Backend registra a tentativa falha e notifica o usuário na interface.

    • O plano anterior é mantido e nenhum limite novo é liberado.

  • A2: Tentativa de Adicionar Sistema acima do Limite do Plano:

    • O Usuário tenta cadastrar um sistema sem ter "slots" disponíveis na assinatura atual.

    • O Sistema bloqueia a ação e redireciona automaticamente para o Fluxo de Upgrade.

OBSERVAÇÕES ARQUITETURAIS:

  • Adapter Pattern (RNF-09): A lógica de "cobrar" é uma porta (BillingPort). Se decidirmos trocar o Mercado Pago pelo Abacate Pay, o Core Domain de assinaturas não é alterado. Apenas criamos um novo AbacatePayAdapter que implementa a mesma interface.

  • Segurança e Idempotência: O sistema deve garantir que o mesmo Webhook de pagamento não seja processado duas vezes (idempotência), evitando que o usuário receba créditos em dobro por engano.

  • Consistência de Dados: O status da assinatura é o "disjuntor" do sistema. Se a assinatura estiver Inativa, os UseCases de "Gerar Tokens" e "Varredura de Logs" devem validar esse estado antes de executar, garantindo a monetização do SaaS.
