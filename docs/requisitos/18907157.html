Minha equipe de software : Caso de uso - 6: Autenticar Usuário com 2FA e Aprovação via Aplicativo Mobile

Created by Michel Bocchi Junior, last modified on mar. 25, 2026

Descrição formal do Caso de uso - 6

Itens associados:

Regras de Negócio:

  • RN-03: A autenticação deve ser obrigatoriamente gerenciada via Clerk

  • RN-14: Para acesso com 2FA via Web, é obrigatória uma validação adicional via aplicativo mobile

  • RN-15: O login Web só é concluído após aprovação explícita no aplicativo

  • RN-16: O sistema deve registrar tentativas rejeitadas como possíveis eventos de segurança

  • RN-17: Solicitações de aprovação devem possuir tempo de expiração

Requisitos Funcionais:

  • RF-34: Suporte à autenticação biométrica

  • RF-35: Suporte à autenticação de dois fatores (2FA)

Requisitos Não Funcionais:

  • RNF-03: Integração com Clerk para autenticação e segurança

Nome: Autenticar Usuário com 2FA Híbrido (Clerk + Aprovação Mobile)

Atores:

  • Usuário

  • Aplicação Web

  • Aplicativo Mobile

  • Sistema LogViewer (Backend)

  • Provedor de Autenticação (Clerk)

Descrição:
Permite autenticar o usuário utilizando o Clerk como provedor de autenticação primária e secundária (2FA padrão), complementado por um segundo fator adicional via aprovação no aplicativo mobile para acessos realizados pela aplicação Web.

Pré-condições:

  • Usuário cadastrado no sistema

  • Usuário autenticável via Clerk

  • Usuário possui 2FA habilitado no Clerk (email ou telefone)

  • Usuário possui aplicativo mobile instalado e vinculado (para acesso via Web)

Pós-condições:

  • Usuário autenticado com sessão válida

  • Evento de autenticação registrado

  • Dispositivo autenticado e validado (quando aplicável)

FLUXO PRINCIPAL 1: LOGIN VIA APLICATIVO MOBILE (2FA GERENCIADO PELO CLERK)

  1. Usuário abre o Aplicativo Mobile

  2. Usuário informa credenciais

  3. Aplicativo Mobile envia requisição ao Backend

  4. Backend redireciona/autentica via Clerk

  5. Clerk valida credenciais

  6. Clerk inicia o fluxo de 2FA padrão:

    • Envio de código via email OU

    • Envio de código via SMS

  7. Usuário informa o código recebido

  8. Clerk valida o segundo fator

  9. Clerk retorna autenticação completa ao Backend

  10. Backend cria sessão autenticada

  11. Backend retorna sucesso ao aplicativo

FLUXO PRINCIPAL 2: LOGIN VIA APLICAÇÃO WEB (2FA + APROVAÇÃO MOBILE)

  1. Usuário acessa a aplicação Web

  2. Usuário informa credenciais

  3. Aplicação Web envia requisição ao Backend

  4. Backend encaminha autenticação ao Clerk

  5. Clerk valida credenciais

  6. Clerk executa o 2FA padrão:

    • Código via email ou SMS

  7. Usuário informa o código

  8. Clerk valida o segundo fator

  9. Backend recebe autenticação válida do Clerk

  10. Backend inicia validação adicional obrigatória:
    10.1 Cria solicitação de aprovação de login
    10.2 Envia notificação push ao Aplicativo Mobile

  11. Aplicação Web exibe status:
    "Aguardando confirmação no aplicativo mobile"

  12. Aplicativo Mobile recebe notificação

  13. Usuário abre o aplicativo

  14. Sistema exibe detalhes do login (ex: localização, horário)

  15. Aplicativo solicita confirmação do usuário:
    15.1 Pode incluir autenticação biométrica

  16. Usuário aprova ou rejeita o acesso

  17. Aplicativo envia resposta ao Backend

  18. Backend valida a resposta:

    • Se aprovado → continua login

    • Se rejeitado → bloqueia acesso

  19. Backend finaliza autenticação

  20. Backend retorna sucesso para aplicação Web

  21. Aplicação Web cria sessão autenticada

FLUXO PRINCIPAL 3: LOGIN SEM 2FA (CASO NÃO ATIVADO NO CLERK)

  1. Usuário informa credenciais

  2. Clerk valida autenticação

  3. Backend cria sessão autenticada

  4. Sistema retorna sucesso

FLUXOS ALTERNATIVOS

A1: Credenciais inválidas

  • Clerk rejeita autentação

  • Sistema retorna erro

A2: Código 2FA inválido (Clerk)

  • Clerk solicita nova tentativa

A3: Usuário não confirma no aplicativo (Web)

  • Solicitação expira

  • Login não é concluído

A4: Usuário rejeita acesso no mobile

  • Backend bloqueia login

  • Registra tentativa suspeita

A5: Aplicativo mobile não disponível (Web)

  • Sistema bloqueia login

  • Informa obrigatoriedade do dispositivo

A6: Falha na entrega da notificação push

  • Backend mantém estado pendente

  • Timeout cancela tentativa

REGRAS DE NEGÓCIO

  • RN-03: Usuários com 2FA ativo devem obrigatoriamente validar o acesso

  • Para acesso com 2FA via Web, é obrigatória uma validação adicional via aplicativo mobile

  • O login Web só é concluído após aprovação explícita no aplicativo

  • O sistema deve registrar tentativas rejeitadas como possíveis eventos de segurança

  • Solicitações de aprovação devem possuir tempo de expiração

REQUISITOS FUNCIONAIS

  • RF-34: Suporte a autenticação biométrica no aplicativo mobile

  • RF-35: Suporte a autenticação de dois fatores (via Clerk)

REQUISITOS NÃO FUNCIONAIS

  • RNF-03: Autenticação gerenciada via Clerk

OBSERVAÇÕES

  • O Clerk é responsável por:

    • Autenticação primária (credenciais)

    • 2FA padrão (email/SMS)

  • O sistema LogViewer adiciona:

    • Segunda camada de segurança para Web (aprovação mobile)

  • O fluxo Web implementa autenticação "out-of-band"

  • O fluxo Mobile utiliza apenas o 2FA do Clerk (sem duplicação de fator)

  • A aprovação mobile deve ser tratada como evento assíncrono (push + callback)

  • Tokens de sessão só devem ser emitidos após:

    • Validação do Clerk

    • (Se Web) aprovação mobile adicional

  • O sistema deve garantir:

    • Idempotência das confirmações

    • Expiração de solicitações pendentes

    • Proteção contra replay de aprovação
