Minha equipe de software : Caso de uso - 8: Apresentar Histórico de Erros

Created by Peng, last modified by João Pedro Souza Peixoto Saraiva on mar. 30, 2026

Nome: Apresentar Histórico de Erros
Atores:

  • Usuário (Gestor/Desenvolvedor)

  • Backend LogViewer

Descrição: O sistema consulta o banco de dados e apresenta ao usuário uma listagem paginada e filtrável de todos os logs de erro capturados, respeitando o isolamento de dados do cliente.

Pré-condições:

  • Usuário autenticado no sistema (Sessão válida via Clerk)

  • Usuário possui ao menos um sistema cadastrado associado ao seu CNPJ

Pós-condições:

  • Lista de erros exibida na interface (Web ou Mobile)

Fluxo Principal:

  1. Usuário acessa a tela de "Histórico de Erros" no Dashboard (Web ou App Mobile)

  2. Sistema frontend solicita a lista de logs ao Backend LogViewer

  3. Backend valida a sessão do usuário e extrai o identificador do CNPJ (Tenant ID)

  4. Backend consulta o MongoDB buscando os logs associados aos sistemas do CNPJ (aplicando paginação)

  5. Backend formata e retorna os dados (Timestamp, Tipo_do_Erro, Status, Criticidade)

  6. Frontend renderiza a lista de histórico para o usuário Fluxos Alternativos:

  • A1: Nenhum erro encontrado

    1. A consulta no MongoDB não retorna documentos.

    2. O sistema exibe um "Empty State" (estado vazio) amigável, informando que não há logs no período ou filtro atual.

  • A2: Usuário utiliza barra de pesquisa ou filtros (RF-21 / RF-14)

    1. Usuário preenche a busca ou seleciona um status específico (ex: "Não Solucionado").

    2. Backend constrói uma query otimizada no MongoDB utilizando índices de texto ou parâmetros exatos.

    3. Sistema retorna e renderiza a lista refinada.

Regras de Negócio:

  • RN-01: Isolamento de dados obrigatório; a busca deve ser sempre restrita ao escopo do CNPJ autenticado.

  • RN-10: O histórico deve responder ao filtro mestre de visão (dados de um "sistema específico" ou "multi-sistemas").

  • RF-23: Os erros apresentados no histórico devem refletir e permitir a distinção de Status (Solucionado, Não solucionado).
