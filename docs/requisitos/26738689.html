Minha equipe de software : Caso de uso - 16: Varredura e Processamento Periódico de Logs (Ingestão em Lote)

Created by Vinicius Ferrarini, last modified on abr. 02, 2026

Itens associados:

Regras de Negócio:

  • RN-04 (Inteligência Contextual e Tipificação): Durante a captura via varredura, o erro extraído do arquivo deve ser classificado e salvo com Timestamp, Tipo_do_Erro, Conteúdo_do_Log e Contexto_Aplicado.

  • RN-05 (Prevenção de Dados Sensíveis): O sistema deve validar ativamente e ofuscar credenciais ou senhas vazadas no conteúdo lido dos arquivos antes da persistência.

Requisitos Funcionais:

  • RF-10: O Sistema deve fazer uma varredura periódica nos logs.

  • RF-07: O Sistema identifica os logs de erros, mesmo que sem padrões.

  • RF-03: O Sistema deve receber o caminho dos Logs da Aplicação configurada (Utilizado como base para a varredura).

Requisitos Não Funcionais:

  • RNF-08: O Sistema deve ser capaz de varrer e processar múltiplos formatos de arquivos de logs.

  • RNF-04: O banco de dados principal será o MongoDB (armazenamento maleável dos dados extraídos).

  • RNF-09: O desenvolvimento deve ser estritamente orientado a Padrões de Projeto (Arquitetura Hexagonal com Strategy Pattern para parseamento de arquivos).

Nome: Varredura e Processamento Periódico de Logs (Ingestão em Lote)

Atores:

  • Sistema Agendador (CronJob / Scheduler) - Ator Primário

  • Sistema LogViewer (Backend)

  • Repositório de Arquivos (Servidor do Cliente)

  • Banco de Dados (MongoDB)

Descrição: Permite que o sistema execute rotinas autônomas (em background) para ler arquivos de log armazenados em diretórios ou servidores configurados pelo cliente. O sistema identifica novos arquivos em múltiplos formatos (ex: .txt, .json, .csv), extrai os dados, ofusca informações sensíveis, tipifica o erro com base no contexto do sistema associado e persiste os registros no banco de dados para futura análise no dashboard.

Pré-condições:

  • O usuário configurou previamente um "Caminho de Origem" (path do diretório ou credenciais de SFTP/S3) atrelado a um de seus sistemas.

  • O sistema possui as permissões necessárias de leitura no diretório alvo.

  • O serviço de Agendamento (Scheduler) do backend está ativo e configurado para o intervalo correto.

Pós-condições:

  • Os erros contidos nos arquivos lidos são normalizados e persistidos no MongoDB.

  • O arquivo original lido é marcado como "processado" (movido, renomeado ou tem seu ponteiro salvo) para evitar ingestão duplicada na próxima varredura.

FLUXO PRINCIPAL 1: EXECUÇÃO DA VARREDURA PERIÓDICA:

  1. O Sistema Agendador (CronJob) dispara o gatilho de execução de acordo com o intervalo configurado (ex: a cada 5 minutos).

  2. O Backend consulta o banco de dados para recuperar a lista de sistemas que possuem "Caminho de Logs" configurado.

  3. Para cada sistema, o Backend acessa o Repositório de Arquivos do Cliente.

  4. O Backend lista os arquivos presentes no diretório e filtra apenas os que ainda não foram processados.

  5. O Backend enfileira os arquivos pendentes para a etapa de processamento.

FLUXO PRINCIPAL 2: PROCESSAMENTO E PERSISTÊNCIA MULTIFORMATO:

  1. O Backend inicia a leitura de um arquivo da fila.

  2. O Backend identifica a extensão/formato do arquivo (ex: .json, .txt).

  3. O Backend aplica o Strategy Pattern no Core Domain, selecionando o Adapter de leitura adequado (JsonParserAdapter, TxtParserAdapter, etc.).

  4. O Adapter correspondente extrai os blocos de texto (erros) e os transforma em um objeto padrão do domínio.

  5. O Backend executa a validação de segurança (RN-05), aplicando Regex ou IA rápida para ofuscar senhas/tokens vazados no texto bruto.

  6. O Backend cruza os dados extraídos com o Contexto configurado (RN-04) e classifica o erro.

  7. O Backend persiste os logs estruturados no MongoDB.

  8. O Backend renomeia o arquivo original (ex: adicionando a extensão .processed) para evitar dupla leitura.

FLUXOS ALTERNATIVOS:

  • A1: Caminho de logs inacessível (Falha de Conexão/Permissão):

    • No Passo 3 do Fluxo 1, o Backend não consegue acessar o diretório configurado.

    • O Backend registra uma falha interna atrelada àquele sistema.

    • O fluxo daquele sistema específico é abortado, mas o Backend continua a varredura para os sistemas dos outros clientes.

    • (Opcional) O sistema gera um alerta no Dashboard do cliente informando que o diretório de logs perdeu a conexão.

  • A2: Formato de arquivo não suportado:

    • No Passo 2 do Fluxo 2, o Backend encontra um arquivo com extensão desconhecida (ex: .dat proprietário).

    • O Strategy Pattern não encontra um Adapter correspondente.

    • O arquivo é ignorado (ou movido para uma pasta de /unsupported), e o sistema passa para o próximo arquivo da fila, sem interromper o serviço.

OBSERVAÇÕES:

  • Arquitetura Hexagonal e Strategy (RNF-08/RNF-09): O domínio não deve conter as regras de "como ler um CSV" ou "como ler um TXT". O caso de uso recebe uma porta LogFileParserPort. Dependendo do arquivo encontrado, a fábrica injeta a estratégia correta (CsvAdapter, TxtAdapter). Isso garante que, se amanhã precisarmos ler logs em XML, basta criar um novo Adapter, sem alterar o motor de ingestão.

  • Performance (Worker Threads): A leitura de arquivos em disco pode bloquear a Event Loop do Node.js (caso a stack backend utilize Node). O processamento dos arquivos (Fluxo 2) deve ser feito utilizando filas assíncronas (ex: RabbitMQ, BullMQ) ou Worker Threads, garantindo que o servidor continue respondendo rapidamente às requisições da aplicação Web/Mobile.

  • Idempotência: A marcação do arquivo como lido (Passo 8 do Fluxo 2) é vital para evitar que o cliente seja tarifado duplamente ou que a inteligência de fadiga de alertas (RN-09) seja disparada equivocadamente por dados velhos lidos de novo.

' Removido o "left to right direction" para empilhar de cima para baixo
skinparam packageStyle rectangle

    BackgroundColor LightCyan

actor "CronJob\n<<Scheduler>>" as Cron
actor "Repositório do Cliente\n<<File System / SFTP>>" as FS
actor "MongoDB\n<<Database>>" as DB
rectangle "LogViewer - Ingestão em Lote" {
usecase "Executar Varredura\\nPeriódica" as UC1
usecase "Listar Arquivos\\nPendentes" as UC2
usecase "Extrair Dados\\n(Strategy Pattern)" as UC3

' Sub-casos (Estratégias)
usecase "Ler arquivo\\nJSON" as UC3_1
usecase "Ler arquivo\\nTXT/CSV" as UC3_2

usecase "Ofuscar Dados\\nSensíveis" as UC4
usecase "Persistir Logs\\nNormalizados" as UC5
}
' Fluxo Inicial (De cima para baixo)
Cron -down-> UC1
' Inclusões do fluxo
UC1 ..> UC2 : <<include>>
UC1 ..> UC3 : <<include>>
' Herança das estratégias empurrando para cima
UC3_1 -up-|> UC3
UC3_2 -up-|> UC3
' Sequência de processamento
UC3 ..> UC4 : <<include>>
UC4 ..> UC5 : <<include>>
' Conexões com os atores de Infraestrutura (Laterais)
FS -right- UC2 : Consulta
FS -right- UC3 : Lê conteúdo
UC5 -left-> FS : Marca '.processed'
UC5 -right-> DB : Salva documento (BSON)
note right of UC1
  RF-10: Rotina autônoma.
end note
note right of UC4
  RN-05: Segurança.
end note
