TR-22 - Criação de caso de uso - 5: Gerenciar Assinatura e Processamento de Pagamento Concluído

Nome: Gerenciar Assinatura e Processamento de Pagamento

Atores:

  • Usuário Administrador

  • Sistema LogViewer

  • Provedor de Autenticação (Clerk)

  • Gateway de Pagamento (externo)

Descrição:
Permite ao usuário administrador contratar, atualizar ou cancelar planos de assinatura, com processamento de pagamento via um gateway externo, mantendo a conta ativa e consistente com os serviços contratados.

Pré-condições:

  • Usuário autenticado via Clerk

  • Conta vinculada a um CNPJ válido

  • Usuário possui permissão de administrador

Pós-condições:

  • Assinatura criada, atualizada ou cancelada

  • Status da conta atualizado (ativo, pendente, inadimplente)

  • Registro de transação armazenado

Fluxo Principal (Contratação ou Atualização de Plano):

  1. Usuário acessa a área de planos e assinatura

  2. Sistema apresenta opções de planos com base na quantidade de sistemas cadastrados

  3. Usuário seleciona um plano ou altera o plano atual

  4. Sistema calcula o valor da assinatura (baseado em quantidade de sistemas)

  5. Sistema solicita confirmação da operação

  6. Sistema envia requisição ao Gateway de Pagamento

  7. Gateway processa o pagamento

  8. Gateway retorna status da transação

  9. Sistema valida o retorno do gateway

  10. Sistema atualiza o status da assinatura:

    • Ativo (pagamento aprovado)

    • Pendente (aguardando confirmação)

  11. Sistema registra a transação

  12. Sistema libera ou atualiza o acesso aos recursos do plano

Fluxos Alternativos:

A1: Pagamento recusado

  • Sistema informa falha ao usuário

  • Mantém plano anterior ou bloqueia ativação

A2: Pagamento pendente (ex: boleto, PIX)

  • Sistema mantém status "pendente"

  • Aguarda confirmação via webhook do gateway

A3: Falha de comunicação com gateway

  • Sistema registra erro

  • Permite nova tentativa de pagamento

A4: Cancelamento de assinatura

  • Usuário solicita cancelamento

  • Sistema atualiza status para "cancelado"

  • Acesso permanece até o fim do ciclo vigente

Fluxo Assíncrono (Webhook de Confirmação):

  1. Gateway envia notificação de pagamento (webhook)

  2. Sistema valida autenticidade do webhook

  3. Sistema atualiza status da assinatura

  4. Sistema registra evento de pagamento confirmado

Regras de Negócio:

  • RN-01: O valor da assinatura deve ser proporcional à quantidade de sistemas cadastrados

  • RN-03: A autenticação do usuário deve ser gerida via Clerk

  • Apenas usuários administradores podem gerenciar assinaturas

  • O sistema deve manter histórico de transações financeiras

Requisitos Funcionais:

  • RF-28: Integração com API de pagamento

Requisitos Não Funcionais:

  • RNF-03: Uso do Clerk para autenticação e segurança

Observações:

  • O sistema deve ser desacoplado do gateway de pagamento, utilizando uma interface (ex: PaymentGatewayPort)

  • A comunicação com o gateway deve suportar múltiplos provedores (ex: estratégia plugável)

  • O processamento de confirmação deve suportar eventos assíncronos (webhooks)

  • Informações sensíveis de pagamento não devem ser armazenadas diretamente no sistema (delegadas ao gateway)

  • O sistema deve estar preparado para múltiplas moedas e modelos de cobrança (mensal, anual)
