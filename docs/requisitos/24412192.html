Minha equipe de software : Caso de uso - 20: Gerir Usuários

Created by Peng, last modified on abr. 02, 2026

Baseado no requisito: RF-15: O Sistema deve gerir os Usuários cadastrados.

Nome: Gerir Usuários

Atores:

  • Administrador do Sistema (Gestor do CNPJ)

  • Usuário Convidado (Desenvolvedor/Analista)

  • Sistema de Autenticação (Clerk)

  • Backend LogViewer

Descrição: Permite que um administrador cadastre, edite, remova ou visualize os usuários que têm acesso aos logs dos sistemas da empresa, controlando níveis de acesso e convites.

Pré-condições:

  • Administrador autenticado com permissões.

  • Empresa (CNPJ) devidamente cadastrada e com plano ativo (RN-01).

Pós-condições:

  • Usuário cadastrado no banco de dados e sincronizado com o Clerk.

  • Permissões de acesso devidamente atribuídas no contexto do sistema.

Fluxo Principal:

  1. O Administrador acessa a seção "Gestão de Usuários" no Dashboard.

  2. O sistema lista todos os usuários vinculados ao CNPJ atual (RN-01).

  3. O Administrador solicita a adição de um novo usuário informando E-mail e Nível de Acesso (Admin ou Dev).

  4. O sistema valida se o plano de assinatura permite a adição de novos usuários (RF-18).

  5. O sistema dispara um convite via Clerk API para o e-mail do convidado.

  6. O Backend LogViewer cria uma entrada de referência no MongoDB vinculando o userID do Clerk ao tenantID (CNPJ).

  7. O Administrador confirma a exclusão ou alteração de nível de um usuário existente.

  8. O sistema sincroniza as alterações instantaneamente com o provedor de identidade.

Fluxos Alternativos:

  • A1: Limite de usuários do plano atingido

    1. O sistema bloqueia a criação do novo usuário.

    2. O sistema sugere o upgrade de plano de assinatura.

  • A2: Usuário já cadastrado em outro CNPJ

    1. O Clerk identifica o e-mail existente.

    2. O sistema vincula o usuário ao novo CNPJ, permitindo que ele alterne entre organizações (Multi-tenancy).

  • A3: Cancelamento de convite pendente

    1. O Administrador revoga o convite antes do aceite.

    2. O sistema invalida o link de acesso via Clerk.

Regras de Negócio Aplicadas:

  • RN-01: Todo usuário criado deve estar obrigatoriamente vinculado a um CNPJ pai.

  • RN-02: A gestão de usuários deve garantir que tokens de SDK sejam visíveis apenas para níveis de acesso autorizados.
