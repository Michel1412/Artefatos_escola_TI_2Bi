Minha equipe de software : Caso de uso - 3: Captura e Identificação de Logs Não Estruturados via SDK

Created by João Pedro Souza Peixoto Saraiva, last modified on mar. 30, 2026

Nome: Captura e Identificação de Logs Não Estruturados via SDK

Atores:

  • Aplicação Cliente (Sistema Gerador do Log)

  • SDK LogViewer (Porta de Entrada/Driving Adapter)

  • Backend LogViewer (Core Domain)

Descrição: Define o propósito e a disponibilidade da SDK do LogViewer como a ferramenta oficial para interceptação de erros na infraestrutura do cliente. A SDK é distribuída publicamente através de gerenciadores de pacotes padrão de mercado (ex: Maven Central para Java/SpringBoot, npm para Node.js, NuGet para ambientes C#). Ela serve exclusivamente para atuar como um "escutador" passivo ou leitor de diretórios, capturando strings de erros, stacktraces e saídas de terminal independentemente de possuírem uma formatação estruturada (como JSON ou XML) ou serem apenas texto livre. Seu papel é empacotar essa saída bruta e enviá-la com segurança para que a Inteligência do LogViewer faça a interpretação.

Pré-condições:

  1. O cliente cadastrou o sistema no painel Web e obteve um token de integração válido.

  2. A SDK do LogViewer foi adicionada como dependência no projeto do cliente (via gerenciador de pacotes).

  3. A SDK foi inicializada no código da aplicação cliente utilizando o token de integração.

  4. A aplicação cliente está em tempo de execução, sujeita a emitir saídas de erro ou exceções não tratadas no console ou em arquivos de log locais.

Pós-condições:

  1. Textos de log (estruturados ou não) são interceptados com sucesso pela SDK.

  2. O log bruto é envelopado em um payload padrão de transporte contendo os metadados mínimos (Timestamp e Token da SDK).

  3. O payload é enviado via túnel criptografado ao Backend do LogViewer e enfileirado para ofuscação (RN-05) e análise do LLM (RN-04).

Fluxo Principal:

  1. A aplicação cliente sofre uma anomalia e emite uma saída de erro arbitrária (ex: uma stacktrace complexa, um print de erro genérico ou uma falha de banco de dados).

  2. A SDK do LogViewer identifica que uma nova saída de erro foi gerada no ecossistema da aplicação.

  3. A SDK captura o bloco de texto do erro de forma integral, sem exigir que o cliente o formate previamente.

  4. A SDK anexa o Timestamp do momento da captura e o token de integração único do sistema ao bloco de texto bruto.

  5. A SDK empacota essas informações em uma requisição leve.

  6. A SDK envia o pacote de forma assíncrona (via webhooks ou tráfego HTTPS criptografado) para a API de ingestão do Backend LogViewer.

  7. O Backend LogViewer recebe o pacote, valida a autenticidade do token e acusa o recebimento com um status de sucesso 202 Accepted.

Fluxos Alternativos:

  • A1: Log excede o limite máximo de tamanho de tráfego

    1. A SDK identifica que o texto capturado é excepcionalmente grande (ex: um dump massivo de memória disfarçado de log).

    2. A SDK trunca o texto até o limite de caracteres permitido, adicionando uma flag [TRUNCATED] ao final da string.

    3. O fluxo retorna ao passo 4 do Fluxo Principal.

  • A2: Aplicação cliente offline ou sem conectividade com o Backend

    1. A SDK tenta enviar o log bruto para o Backend LogViewer e falha por timeout de rede.

    2. A SDK armazena o log de forma temporária e volátil na memória ou disco local do cliente (estratégia de buffer).

    3. A SDK implementa um mecanismo de retry com backoff exponencial até que a conexão seja restabelecida para esvaziar o buffer.

Regras de Negócio:

  • RN-04: Inteligência Contextual e Tipificação

  • RN-05: Prevenção de Dados Sensíveis

  • RN-08: Notificação Multicanal Obrigatória

Requisitos Funcionais:

  • RF-07: O Sistema identifica os logs de erros, mesmo que sem padrões

  • RF-12: O Sistema deve disponibilizar uma SDK capaz de capturar e enviar os logs da aplicação do cliente

Requisitos Não Funcionais:

  • RNF-01: Comunicação criptografada obrigatória (TLS/Túnel seguro)

  • RNF-02: Baixo impacto na performance

Observações:

  • Abstração Arquitetural: Dentro da Arquitetura Hexagonal, a SDK age estritamente fora do hexágono. Ela não possui inteligência de categorização de erros nem capacidade de entender o "Contexto" da empresa; seu escopo é puramente a extração agnóstica de texto e o transporte seguro. Toda a carga de "entender" o erro sem padrão fica delegada ao Core Domain (com apoio do LLM) no Backend.

  • Agnosticismo de Padrão: A documentação fornecida aos desenvolvedores clientes deve deixar explícito que não é necessário criar parsers customizados ou formatar os erros em JSON antes de enviá-los à SDK. A promessa de valor do RF-07 é justamente a fricção zero na ingestão.
