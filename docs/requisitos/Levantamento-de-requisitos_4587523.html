Minha equipe de software : Levantamento de requisitos

Created by Michel Bocchi Junior, last modified on mar. 12, 2026

Requisitos Funcionais

  • RF - 01: O Sistema deve autenticar o Usuário ``

  • RF - 02: O Sistema pergunta ao Usuário se deseja uma solução para o erro descoberto

  • RF - 03: O Sistema deve receber o caminho dos Logs da Aplicação configurada

  • RF - 04: O Sistema deve notificar os erros encontrados na Varredura

  • RF - 05: O Sistema deve alertar possíveis situações criticas descobertas pelo SDK

  • RF - 06: O Sistema deve apresentar um histórico de erros

  • RF - 07: O Sistema identifica os logs de erros, mesmo que sem padrões

  • RF - 08: O Usuário deve configurar um Contexto onde sua empresa está atuando

  • RF - 09: O Sistema aplicará o contexto configurado no momento da Varredura

  • RF - 10: O Sistema deve fazer uma varredura periódica nos logs

  • RF - 11: O Usuário poderá fazer a alteração do contexto

  • RF - 12: O Sistema deve disponibilizar uma SDK capaz de capturar e enviar os logs da aplicação do cliente

  • RF - 13: O Sistema deve compartilhar o link do Agente, que recebeu o contexto e os possíveis logs de erro

  • RF - 14: O Sistema deve filtrar logs por várias opções

  • RF - 15: O Sistema deve gerir o Usuário

  • RF - 16: O Sistema deve gerencias tokens para a integração SDK.

  • RF - 17: O Sistema deve ter um Painel com Dashboard.

  • RF - 18: O Sistema deve conter um gerenciamento de assinatura do usuário

  • RF - 19: O Sistema vai sugerir soluções para problemas recorrentes

  • RF - 20: O Sistema deve conter uma página de relatório para os multi-sistemas

  • RF - 21: O Sistema deve ter uma barra de pesquisa dos erros encontrado já armazenado

  • RF - 22: O Sistema tem que ter um contador de logs repetidos

  • RF - 23: O Sistema tem que ter a opção Status por erro Não solucionado, Solucionado

  • RF - 24: O Sistema tem que ter a opção de exportar para PDF e Excel

  • RF - 25: O Sistema tem que validar se tem senhas vazadas nos logs

  • RF - 26: O Sistema tem que existir uma maneira de confirmação de leitura

  • RF - 27: O Sistema deve poder arquivar erros

  • RF - 28: O Sistema deve ter uma integração com uma API de pagamento (ex: Mercado Pago, abacate pay)

  • RF - 29: O Sistema deve ter uma interface mobile

  • RF - 30: O aplicativo mobile deve notificar falhas

  • RF - 31: O aplicativo deve facilitar a comunicação com o LLM

  • RF - 32: O aplicativo deve ter um espelho do dashboard do sistema web

  • RF - 33: O aplicativo deve ser lançado em ambas as plataformas mobile (Android e IOS)

  • RF - 34: O aplicativo deve conter um autenticador via face Id ou touch Id

  • RF - 35: O aplicativo deve apoiar a autenticação de dois fatores

Requisitos Não Funcionais

  • RN - 01: A conexão do SDK com o Core deve ser criptografada.

  • RN - 02: A primeira versão do SDK deve ser em JAVA

  • RN - 03: O projeto deve estimar outras versão de SDKs

  • RN - 04: Debater sobre utilizar um Sistema externo de gestão de usuários

  • RN - 05: Debater sobre o tipo de Banco de dados

  • RN - 06: Escolher qual banco de dados atende melhor às necessidades

  • RN - 08: Escolher a arquitetura do Sistema.

  • RN - 09: O Sistema deve salvar logs temporariamente

  • RN - 10: Disponibilizar um Webhook para os SDKs

  • RN - 11: O Sistema deve ser capaz de varrer qualquer arquivo que armazenam historicos de Logs.

  • RN - 12: Utilizar padrões de projeto para melhorar a qualidade e eficiência

  • RN - 13: O aplicativo e o sitema web devem utilizar dados otimizados, visando reduzir o PLT (Page Load Time)
