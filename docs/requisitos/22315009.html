Minha equipe de software : Caso de uso - 15: Confirmação de Leitura de Logs no Dashboard

Created by Vinicius Ferrarini, last modified on mar. 30, 2026

TR-32 - Criação de caso de uso - 15 Gerenciar Alertas e Confirmação de Leitura de Notificações Concluído

Itens associados:

Regras de Negócios:

  • RN-10: Visões do Dashboard. O Dashboard deve exibir dados consolidados e permitir a diferenciação visual e filtragem baseada no status de leitura do erro.

  • RN-12: Gestão de Marcadores. O sistema deve permitir a organização da "caixa de erros", utilizando o estado intrínseco de "Lido/Não Lido" como um marcador padrão do sistema para facilitar a triagem operacional. Requisitos Funcionais:

Requisitos Funcionais:

  • RF-26: O Sistema deve possuir um mecanismo de confirmação de leitura informando se o log encontrado já foi acessado no dashboard.

  • RF-14: O Sistema deve permitir filtrar logs por múltiplas opções (Apoiado por este fluxo, permitindo filtrar por "Lidos" e "Não Lidos").

Requisitos Não Funcionais:

  • RNF-04: O banco de dados principal será o MongoDB.

  • RNF-09: O desenvolvimento deve ser estritamente orientado a Padrões de Projeto (Arquitetura Hexagonal).

Nome: Confirmação de Leitura de Logs no Dashboard

Atores:

  • Usuário (Via Web ou Mobile)

  • Aplicação Web (React) / Aplicativo Mobile (React Native)

  • Sistema LogViewer (Backend)

  • Banco de Dados (MongoDB)

Descrição:

Garante que o Dashboard funcione de maneira semelhante a uma "caixa de entrada" de e-mails. O sistema deve registrar e indicar visualmente quando um log de erro inédito foi aberto e visualizado pelo usuário, atualizando seu estado para "Lido". Isso facilita a gestão operacional, permitindo que a equipe saiba rapidamente quais erros já foram investigados e quais ainda precisam de atenção.

Pré-condições:

  • Usuário autenticado e navegando na interface do Dashboard (Web ou Mobile).

  • O banco de dados possui logs registrados para os sistemas associados ao CNPJ do usuário.

Pós-condições:

  • O documento do Log no MongoDB tem sua propriedade de controle atualizada (ex: isRead: true).

  • A interface do usuário é atualizada em tempo real para remover o destaque visual de "Não Lido" daquele item.

FLUXO PRINCIPAL 1: ACESSO E CONFIRMAÇÃO DE LEITURA (RF-26):

  1. Usuário acessa a listagem de erros no Dashboard.

  2. Sistema (Frontend) renderiza a lista, aplicando um destaque visual (ex: texto em negrito, ícone de badge ou cor de fundo diferente) em todos os logs que possuem o status de leitura pendente (isRead: false).

  3. Usuário clica em um log destacado para abrir a tela/modal de Detalhes do Erro.

  4. Sistema (Frontend) exibe as informações completas do erro (Payload, Contexto, etc.).

  5. Sistema (Frontend) dispara, em background, uma requisição assíncrona para o Backend sinalizando a leitura (Ex: PATCH /logs/{id}/read).

  6. Backend recebe a requisição e atualiza o documento específico no MongoDB, definindo isRead: true.

  7. Backend retorna status de sucesso (200 OK) para o Frontend.

  8. Sistema (Frontend) atualiza o estado local (React State) e remove o destaque visual de "Não Lido" daquele log na listagem.

FLUXOS ALTERNATIVOS:

  • A1: Log já havia sido lido anteriormente:

    • No Passo 3, o Usuário clica em um log que já foi acessado no passado (não possui destaque visual).

    • O Sistema (Frontend) abre os detalhes do log normalmente.

    • O Sistema identifica localmente que o log já está como isRead: true e aborta o Passo 5 (não faz a requisição de atualização para a API).

    • O fluxo é encerrado, economizando tráfego de rede e processamento do servidor.

  • A2: Falha na comunicação com o Backend ao confirmar leitura:

    • No Passo 5, a requisição assíncrona falha (timeout ou erro 500).

    • O Sistema (Frontend) loga o erro silenciosamente no console.

    • O usuário continua vendo os detalhes do erro normalmente (não é bloqueado), mas o log pode voltar a aparecer como "Não Lido" em um próximo refresh da página, garantindo a resiliência dos dados.

OBSERVAÇÕES ARQUITETURAIS:

  • Modelagem MongoDB (RNF-04): A entidade Log no banco de dados ganha um campo booleano nativo. O schema básico deve contemplar isRead: { type: Boolean, default: false }. Isso garante que todo log recém-capturado nasça automaticamente como não lido.

  • Filtros do Dashboard (RF-14): Com a implementação deste caso de uso, a barra de filtros do Dashboard ganha a capacidade técnica de buscar por status de visualização. O backend deve aceitar a flag isRead como parâmetro de query na rota de listagem (ex: GET /logs?isRead=false).

  • Baixo PLT / Otimização UX: A mudança de estado visual na interface (Passo 8) deve ocorrer através de Optimistic UI Update (atualização otimista). O React/React Native altera a interface no instante do clique, sem obrigar o usuário a esperar o retorno da API, garantindo uma sensação de extrema fluidez.
