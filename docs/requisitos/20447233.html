Minha equipe de software : Caso de uso - 14: Gerar e Exportar Relatórios Multi-sistemas

Created by Vinicius Ferrarini, last modified on mar. 30, 2026

TR-31 - Caso de Uso - 14: Gerar e Exportar Relatórios Multi-sistemas Concluído

Itens associados:

Regras de Negócio:

  • RN-01: Identidade e Precificação. Todo usuário base é um CNPJ, podendo ele ter múltiplos sistemas cadastrados debaixo deste identificador.    

  • RN-10: Visões do Dashboard. O Dashboard deve exibir dados consolidados de todos os sistemas, possuindo um filtro mestre para visualizar o agregado de "multi-sistemas".    

  • RN-11: Busca e Exportação. Os relatórios gerados a partir do histórico consolidado devem poder ser exportados nativamente para PDF e Excel.

Requisitos Funcionais:

  • RF-20: O Sistema deve conter uma página de relatório unificada para multi-sistemas.    

  • RF-24: O Sistema deve exportar relatórios para PDF e Excel.

Requisitos Não Funcionais:

  • RNF-04: O banco de dados principal será o MongoDB (utilização de Aggregation Pipelines para consolidar os dados dos múltiplos sistemas).    

  • RNF-09: O desenvolvimento deve ser estritamente orientado a Padrões de Projeto (Arquitetura Hexagonal com Strategy Pattern para exportação).

Nome: Gerar e Exportar Relatórios Multi-sistemas

Atores:

  • Usuário (Gestor / Administrador)

  • Aplicação Web / Aplicativo Mobile

  • Sistema LogViewer (Backend)

  • Banco de Dados (MongoDB)

Descrição:

Permite ao usuário acessar uma visão consolidada de todos os erros de diferentes sistemas atrelados ao seu CNPJ em uma única interface. O usuário pode analisar os dados agregados e solicitar a exportação desse relatório estruturado nos formatos PDF ou Excel para análises externas ou apresentações.

Pré-condições:

  • Usuário autenticado no sistema (Web ou Mobile).

  • O CNPJ do usuário possui pelo menos um (idealmente múltiplos) sistemas cadastrados.

  • Existem logs armazenados no MongoDB vinculados aos sistemas do CNPJ do usuário.

Pós-condições:

  • Os dados agregados multi-sistemas são renderizados em tela (gráficos e tabelas).

  • Um arquivo digital (PDF ou Excel) contendo o relatório é gerado pelo servidor e disponibilizado para download pelo usuário.

FLUXO PRINCIPAL 1: VISUALIZAÇÃO DO RELATÓRIO UNIFICADO:

  • Usuário acessa a aba de "Relatórios Consolidados" (Multi-sistemas).    

  • Sistema (Frontend) solicita ao Backend os dados analíticos referentes ao CNPJ do usuário.

  • Backend executa uma Aggregation Pipeline no MongoDB agrupando logs por sistema, status e criticidade.    

  • Backend retorna o conjunto de dados consolidados e sumarizados.

  • Sistema exibe os gráficos, totalizadores e tabelas comparativas entre os sistemas do usuário.

FLUXO PRINCIPAL 2: EXPORTAÇÃO DE RELATÓRIO (PDF/EXCEL):

  • Usuário visualiza a página de relatório unificada e aplica filtros desejados (ex: últimos 30 dias).

  • Usuário clica no botão Exportar Relatório.

  • Sistema exibe as opções de formato: "PDF" ou "Excel".

  • Usuário seleciona o formato desejado (ex: PDF)

  • Sistema (Frontend) envia a requisição de exportação ao Backend contendo os filtros e o formato desejado

  • Backend valida a requisição e busca os dados brutos no MongoDB baseados nos filtros

  • Backend aplica o Strategy Pattern no Core Domain para selecionar o Adapter correto (PdfExportAdapter ou ExcelExportAdapter).

  • O Adapter correspondente processa os dados e gera um buffer de arquivo na memória do servidor.

  • Backend retorna o arquivo gerado via stream ou link temporário seguro para o Frontend.

  • Sistema realiza o download automático do arquivo no dispositivo do Usuário

FLUXOS ALTERNATIVOS:

  • A1: Nenhum dado disponível para exportação:

    • Usuário tenta exportar um relatório de um período sem logs registrados.

    • Backend identifica a ausência de dados antes de acionar o Adapter de exportação.

    • Sistema exibe a mensagem: "Não há dados suficientes neste período para gerar o relatório."

  • A2: Falha na geração do arquivo por volume de dados excessivo:

    • Backend tenta alocar em memória um relatório Excel gigante e atinge o limite (timeout ou out-of-memory).

    • Backend cancela a operação e retorna um erro 500 tratado.

    • Sistema exibe interface de erro recomendando que o usuário diminua o intervalo de datas do filtro.

OBSERVAÇÕES:

A geração de arquivos (PDF/Excel) ocorre estritamente no Backend. Isso é uma decisão arquitetural para manter o Baixo PLT (Page Load Time) exigido para a aplicação React Web e evitar travamentos de interface no React Native (Mobile), visto que bibliotecas de renderização de PDF no client-side são pesadas.     O uso do Strategy Pattern facilita a extensibilidade do sistema. Se no futuro o LogViewer precisar exportar relatórios para CSV ou JSON nativo, bastará criar um novo Adapter, sem alterar absolutamente nenhuma linha de código da regra de negócio (Core Domain).     A autenticação do Driving Adapter (rota da API) repassa as credenciais baseadas no Clerk, garantindo que o usuário só baixe logs do seu próprio CNPJ.
