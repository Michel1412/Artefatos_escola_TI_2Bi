Minha equipe de software : Equipe 3

Created by João Pedro Souza Peixoto Saraiva, last modified by pedroalvaro193 on mar. 12, 2026

  • RF - 01: O Sistema deve autenticar o Usuário

  • RF - 02: O Sistema pergunta ao Usuário se deseja uma solução para o erro descoberto

  • RF - 03: O Sistema deve receber o caminho dos Logs da Aplicação configurada

  • RF - 04: O Sistema deve notificar os erros encontrados na Varredura

  • RF - 05: O Sistema deve alertar possíveis situações criticas descobertas pelo SDK

  • RF - 06: O Sistema deve apresentar um histórico de erros

  • RF - 07: O Sistema deve ser capaz de classificar entradas de log como "Erros" utilizando Processamento de Linguagem Natural (NLP), independentemente da formatação (ex: JSON, Plain Text, XML).

  • RF - 08: O Usuário deve configurar um Contexto onde sua empresa está atuando

  • RF - 09: O Sistema aplicará o contexto configurado no momento da Varredura

  • RF - 10: O Sistema deve fazer uma varredura periódica nos logs

  • RF - 11: O Usuário poderá fazer a alteração do contexto

  • RF - 12: O Sistema deve disponibilizar uma SDK capaz de capturar e enviar os logs da aplicação do cliente

  • RF - 13: O Sistema deve compartilhar o link do Agente, que recebeu o contexto e os possíveis logs de erro

  • RF - 14: O Sistema deve permitir a filtragem do histórico de logs pelos critérios de: Data/Hora, Severidade (Warning, Error, Fatal), Ambiente (Contexto) e Palavra-chave.
    =======================================

  • RN - 01 - O sistema deve APENAS perguntar se o user deseja receber uma solução. (RF - 02, RF - 13)
    Ex: Fulano, voce deseja receber uma possivel solucão para o erro 23456789 encontrado?

O sistema não deve aplicar correções automaticamente. Toda solução gerada pela IA deve passar por uma etapa de consentimento do usuário.

Fluxo (Sugestão Ativa): O sistema apresenta a solução A e questiona a aceitação, e disponibiliza o link do agente que recebeu o erro e o contexto aplacado o encaminhando para a página de conversa do agente.

Possiveis variações
O sistema sugere uma solução e pergunta se ele quer acatar ou procurar outra possivel solução
Ex: Fulano, essa solucao (limpar cache) esta do seu agrado? voce deseja prosseguir com ela ou quer analisar outras possibilidades?

  • RN - 02 - Armazenamento e notificação de erro (RF - 04, RF - 05, RF - 06)

Ao iniciar varredura o sistema deve identificar o tipo de erros, armazenar em um banco contendo data, tipo, conteúdo e contexto aplicado.

Todo erro identificado na varredura deve seguir obrigatoriamente o fluxo: Identificação → Persistência → Alerta.

Persistência: O registro no banco de dados deve conter obrigatoriamente: Timestamp, Tipo_do_Erro, Conteúdo_do_Log e Contexto_Aplicado.

Notificação Multicanal:

  • Assíncrona: Envio de E-mail e Push Notification no App Mobile.

  • Síncrona: Se o usuário estiver com o Dashboard web aberto (via WebSockets), um alerta visual (toast/modal) deve aparecer em tempo real.

Regra de Silenciamento: Caso o mesmo erro ocorra repetidamente em um intervalo de 5 minutos, as notificações de e-mail devem ser agrupadas em um único resumo para evitar fadiga de alertas.

  • RN - 03: Varredura de Logs com Inteligência Contextual

Fluxo de Execução:

  1. Input: O sistema captura os logs (via SDK ou caminho configurado).

  2. Processamento (IA + Contexto): A IA cruza o log bruto com as respostas da Pesquisa de Onboarding (Ramo, Stack, Criticidade) realizada após o pagamento.

  3. Classificação: O erro é tipificado (Ex: Erro de Checkout em E-commerce = Crítico; Erro de CSS em Blog = Baixo).

  4. Persistência: O log é salvo no banco com: Data, Tipo, Conteúdo e o Contexto Aplicado.

  5. Notificação Multicanal:

    • Dashboard: Alerta visual em tempo real (se logado).

    • E-mail/App: Push automático com o resumo do erro e link para a solução.



$$ Ideia para modelo de negócio:
Crie mais um plano, onde o cliente ele paga mais. para receber uma solução objetiva ao encontrar um erro.
