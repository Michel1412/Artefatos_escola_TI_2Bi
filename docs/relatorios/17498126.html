Minha equipe de software : TaskList - Requisitos > Caso de Uso

Created by João Pedro Souza Peixoto Saraiva, last modified by Vinicius Ferrarini on abr. 02, 2026

Requisitos Funcionais (RF)

Ações e funcionalidades diretas que o sistema deve executar.

  • RF-01: O Sistema deve autenticar o Usuário. @peng
  • RF-02: O Sistema deve perguntar ao Usuário se deseja uma solução para o erro descoberto. pedroalvaro193
  • RF-03: O Sistema deve receber o caminho da pasta de Logs da Aplicação configurada. pedroalvaro193
  • RF-04: O Sistema deve notificar os erros encontrados na varredura. João Pedro Souza Peixoto Saraiva
  • RF-05: O Sistema deve alertar possíveis situações críticas descobertas pelo SDK. João Pedro Souza Peixoto Saraiva
  • RF-06: O Sistema deve apresentar um histórico de erros. Peng
  • RF-07: O Sistema deve identificar os logs de erros, mesmo que não possuam padrões estruturados. João Pedro Souza Peixoto Saraiva
  • RF-08: O Usuário deve configurar um Contexto detalhando a área de atuação de sua empresa. pedroalvaro193
  • RF-09: O Sistema aplicará o contexto configurado no momento da varredura. pedroalvaro193
  • RF-10: O Sistema deve fazer uma varredura periódica nos logs. Lauro Neves (Unlicensed)
  • RF-11: O Usuário poderá alterar o contexto configurado a qualquer momento.pedroalvaro193
  • RF-12: O Sistema deve disponibilizar uma SDK capaz de capturar e enviar os logs da aplicação do cliente. João Pedro Souza Peixoto Saraiva
  • RF-13: O Sistema deve compartilhar o link do Agente LLM, contendo o contexto e os logs de erro para interação. Peng
  • RF-14: O Sistema deve permitir filtrar logs por múltiplas opções (ex: data, criticidade, sistema). Vinicius Ferrarini
  • RF-15: O Sistema deve gerir os Usuários cadastrados. Peng
  • RF-16: O Sistema deve gerenciar os tokens de integração da SDK. Michel Bocchi Junior
  • RF-17: O Sistema deve possuir um Painel (Dashboard) interativo. Peng
  • RF-18: O Sistema deve conter um gerenciamento de assinatura/planos do usuário. Lauro Neves (Unlicensed)
  • RF-19: O Sistema deve sugerir soluções para problemas recorrentes. pedroalvaro193
  • RF-20: O Sistema deve conter uma página de relatório unificada para multi-sistemas. Vinicius Ferrarini
  • RF-21: O Sistema deve ter uma barra de pesquisa global para os erros já armazenados. Vinicius Ferrarini
  • RF-22: O Sistema deve ter um contador/agrupador de logs repetidos. pedroalvaro193
  • RF-23: O Sistema deve categorizar erros por Status (Ex: Não Solucionado, Solucionado). Vinicius Ferrarini
  • RF-24: O Sistema deve exportar relatórios para PDF e Excel. Vinicius Ferrarini
  • RF-25: O Sistema deve validar e mascarar se há senhas/credenciais vazadas nos logs. Michel Bocchi Junior
  • RF-26: O Sistema deve possuir um mecanismo de confirmação de leitura das notificações. Vinicius Ferrarini
  • RF-27: O Sistema deve permitir o arquivamento de erros. pedroalvaro193
  • RF-28: O Sistema deve ser integrado a uma API de pagamento (ex: Mercado Pago, Abacate Pay). Michel Bocchi Junior
  • RF-29: O Sistema deve possuir uma interface mobile (aplicativo). João Pedro Souza Peixoto Saraiva
  • RF-30: O aplicativo mobile deve notificar falhas via push. João Pedro Souza Peixoto Saraiva
  • RF-31: O aplicativo deve conter um chat integrado para facilitar a comunicação com o LLM. Michel Bocchi Junior
  • RF-32: O aplicativo deve possuir um espelho simplificado do dashboard do sistema web. João Pedro Souza Peixoto Saraiva
  • RF-33: O aplicativo deve ser lançado para Android e iOS. João Pedro Souza Peixoto Saraiva
  • RF-34: O aplicativo deve suportar autenticação biométrica (Face ID / Touch ID). Michel Bocchi Junior
  • RF-35: O aplicativo deve suportar e apoiar a Autenticação de Dois Fatores (2FA). Michel Bocchi Junior
  • RF-36: O Sistema despacha um evento de E-mail, detalhando os erros críticos encontrados na varredura João Pedro Souza Peixoto Saraiva

Requisitos Não Funcionais (RNF)

Critérios de qualidade, arquitetura, desempenho e segurança.

  • RNF-01: A conexão entre a SDK e o Core deve ser obrigatoriamente criptografada (Túnel de Criptografia). Michel Bocchi Junior
  • RNF-02: A primeira versão da SDK deve ser desenvolvida em Java, com previsão para expansão a outras linguagens. Peng
  • RNF-03: O Sistema utilizará a plataforma externa Clerk para gestão de usuários, garantindo maior segurança e escalabilidade. Michel Bocchi Junior
  • RNF-04: O banco de dados principal será o MongoDB, visando maleabilidade no armazenamento de logs não estruturados. Vinicius Ferrarini
  • RNF-05: O Sistema adotará a Arquitetura Hexagonal (Ports and Adapters). Michel Bocchi Junior
  • RNF-06: O Sistema deve salvar logs temporariamente (retenção de 7 dias dependendo do plano). Vinicius Ferrarini
  • RNF-07: A API de recebimento deve ser focada em alta performance e utilizar Webhooks para evitar travamento na aplicação cliente (SDK). Michel Bocchi Junior
  • RNF-08: O Sistema deve ser capaz de varrer e processar múltiplos formatos de arquivos de logs. Lauro Neves (Unlicensed)
  • RNF-09: O desenvolvimento deve ser estritamente orientado a Padrões de Projeto (Design Patterns). Vinicius Ferrarini
  • RNF-10: O frontend Web será desenvolvido em React (ou React Native Web), visando otimização e padronização da equipe. (Nota: Vi que sugeriu React Native para web; é válido se usarem bibliotecas de unificação, embora React JS seja o padrão. É uma escolha interessante considerando que sua bagagem mobile anterior era forte em Flutter!) Peng
  • RNF-11: O aplicativo Mobile será desenvolvido em React Native. Lauro Neves (Unlicensed)
  • RNF-12: O aplicativo e o sistema web devem utilizar paginação e dados otimizados para reduzir o PLT (Page Load Time). Lauro Neves (Unlicensed)
  • RNF-13: Os repositórios e a gestão das equipes de Mobile e Web devem ser separados. Lauro Neves (Unlicensed)
