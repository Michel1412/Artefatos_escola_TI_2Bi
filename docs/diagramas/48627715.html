Minha equipe de software : Diagrama de Sequência 7-> Caso de uso 7: O Sistema deve autenticar o Usuário

Created by Vinicius Ferrarini, last modified by Michel Bocchi Junior on mai. 18, 2026

TR-56 - Criação diagrama de sequencia 7 Concluído

  • Passo 1: Comunicação do nosso backend com o serviço externo (Clerk).

  • Passo 2: Comunicação assíncrona do backend com o celular do cliente (Push).

  • Passo 3: Comunicação interna do nosso backend com o nosso próprio Banco de Dados (MongoDB) validando regras financeiras.

Parte 1: Validação de Credenciais (Clerk)

Focado apenas na tentativa inicial do usuário de colocar e-mail e senha.

@startuml
!theme plain
autonumber
skinparam maxMessageSize 150
skinparam sequence {
    ParticipantBackgroundColor #EBF8FF
    ParticipantBorderColor #005A9C
    ActorBorderColor #005A9C
}

title 1. Validação de Credenciais (Clerk)

actor "Usuário" as User
participant "Frontend\n(Web/Mobile)" as UI
participant "Backend API" as API
participant "Clerk Auth" as Clerk

User -> UI: Insere credenciais (E-mail/Senha)
activate UI
UI -> API: Inicia requisição de Login
activate API
API -> Clerk: Validar credenciais
activate Clerk

alt Exceção A1: Credenciais Inválidas
    Clerk --> API: Retorna falha de autenticação
    API --> UI: Retorna erro 401 (Não autorizado)
    UI --> User: Exibe: "E-mail ou senha incorretos"
else Credenciais Válidas
    Clerk --> API: Sucesso (Retorna dados básicos)
end
deactivate Clerk
deactivate API
deactivate UI
@enduml

Parte 2: Verificação de Segundo Fator (App Mobile)

Focado no momento em que a senha está certa, mas o sistema exige a confirmação biométrica pelo celular.

@startuml
!theme plain
autonumber
skinparam maxMessageSize 150
skinparam sequence {
    ParticipantBackgroundColor #EBF8FF
    ParticipantBorderColor #005A9C
    ActorBorderColor #005A9C
}

title 2. Verificação de Segundo Fator (MFA)

actor "Usuário" as User
participant "Frontend\n(Web/Mobile)" as UI
participant "Backend API" as API
participant "App Mobile\n(Aprovador)" as App

note over API: Senha validada no Passo 1
activate API

API -> API: Verifica se perfil exige 2FA

alt Exceção A2: 2FA Ativo mas App não instalado
    API --> UI: Retorna erro 403 (Ação bloqueada)
    activate UI
    UI --> User: Exibe alerta: "Instale o app para validar o acesso"
    deactivate UI
    
else App Mobile Instalado
    API -> App: Dispara Notificação Push
    activate App
    App --> User: Solicita aprovação (Face ID/Touch ID)
    User -> App: Aprova acesso biometricamente
    App -> API: Retorna confirmação de 2FA aprovado
    deactivate App
end
deactivate API
@enduml

Parte 3: Validação de Assinatura e Liberação

Focado na regra de negócio final: checar se a conta está paga no banco de dados e liberar o acesso.

@startuml
!theme plain
autonumber
skinparam maxMessageSize 150
skinparam sequence {
    ParticipantBackgroundColor #EBF8FF
    ParticipantBorderColor #005A9C
    ActorBorderColor #005A9C
    DatabaseBackgroundColor #EBF8FF
    DatabaseBorderColor #005A9C
}

title 3. Validação de Assinatura e Liberação

actor "Usuário" as User
participant "Frontend\n(Web/Mobile)" as UI
participant "Backend API" as API
database "Banco de Dados" as DB

note over API: 2FA aprovado no Passo 2
activate API

API -> DB: Consultar status do CNPJ vinculado
activate DB
DB --> API: Retorna status da assinatura
deactivate DB

alt Exceção A3: Assinatura Inativa / Inadimplente
    API -> API: Gera token restrito
    API --> UI: Redireciona para área financeira
    activate UI
    UI --> User: Exibe página de Gestão de Assinatura
    deactivate UI
    
else Assinatura Ativa
    API -> API: Gera JWT final
    API --> UI: Libera acesso ao sistema
    activate UI
    UI --> User: Exibe Dashboard principal do LogViewer
    deactivate UI
end
deactivate API
@enduml
